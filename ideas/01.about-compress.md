# About Context Compression in AI Agents

## The Core Problem

The context window is finite. Every tool call result (file contents, bash output) eats tokens.
After reading 30 files and running 20 commands, you easily hit 100,000+ tokens.
The agent slows down, gets expensive, and eventually hits the hard limit.

## How micro_compact Works

`micro_compact` keeps the last N tool results intact and replaces older ones with one-line placeholders:

```
Before:
  tool_result #1: "(3000 chars of file content)"     <- old
  tool_result #2: "(500 chars of ls output)"          <- old
  tool_result #3: "Wrote 128 bytes"                   <- recent

After:
  tool_result #1: "[Previous: used read_file]"        <- compressed
  tool_result #2: "[Previous: used bash]"             <- compressed
  tool_result #3: "Wrote 128 bytes"                   <- kept
```

Only `tool_result` blocks (in user messages) are compressed.
Assistant messages (tool_use blocks and text) are **never** touched.

## Key Insight: AI Has No Intermediate Storage

The AI does NOT work like this (what you might expect):

```
Read file1 -> analyze -> store "file1 is about X" ->
Read file2 -> analyze -> store "file2 is about Y" ->
...
Combine all stored analyses -> output final summary
```

The AI actually works like this:

```
Round 1: "I need to list files"    -> [tool_use: bash ls]         (no analysis text)
Round 2: "I need to read files"    -> [tool_use: read_file x7]    (no analysis text)
Round 3: "Some results compressed" -> [tool_use: bash cat x4]     (no analysis text)
Round 4: "Read the rest"           -> [tool_use: bash cat x3]     (no analysis text)
Round 5: "All done"                -> [text: "Here's the summary of all 7 files..."]
```

In rounds 2-4, the AI only emits tool calls with **zero analysis text**.
All thinking and summarizing happens in one shot at the final round.

## The Consequence

When the AI generates the final summary, it looks back at `messages` to recall file contents.
But old `tool_result` entries have been replaced with `"[Previous: used read_file]"`.

So for compressed files, the AI relies on:
1. **File names** — `s01_agent_loop.py` is self-descriptive
2. **tool_use history** — assistant messages show what commands were run (e.g. `bash(command="cat agents/s01_agent_loop.py")`)
3. **Training data** — the AI knows what "agent loop" or "tool use" means
4. **Inference from recent results** — the 3 most recent files are intact, providing context

This means summaries of compressed files are **less accurate** than summaries of recent files.

## The Trade-off

| | Without compression | With compression |
|---|---|---|
| Token usage | High (hits limit fast) | Low (can keep working) |
| Accuracy | Full recall of all results | Recent results accurate, old results approximate |
| Cost | Expensive | Cheaper |
| Side effect | None | AI may re-read files it already read |

## Possible Improvement

To avoid the "AI forgets old results" problem, you could instruct the AI
in the system prompt to summarize each file as it reads it:

> "After reading each file, write a brief summary of your findings before continuing."

This way each file's analysis is saved as assistant **text** (which micro_compact never touches),
rather than being trapped in a tool_result (which gets compressed).

## Why s07 (Task System) Persists to Disk

State stored in `messages` will be lost to compression.
State stored on the filesystem (`.tasks/*.json`) survives forever.

This is the key architectural lesson:
**Anything the agent needs to remember long-term should be written to disk, not kept in conversation history.**
