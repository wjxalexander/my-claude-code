# Threading, Subprocess, and Background Tasks

## Python Threading

`threading` 模块在同一个进程内创建多个线程，共享内存。

核心限制：CPython 有 GIL（Global Interpreter Lock），同一时刻只有一个线程能执行 Python 字节码。
- I/O 密集型任务（网络、文件）：线程有效，因为等待 I/O 时释放 GIL
- CPU 密集型任务（纯计算）：线程无效，应改用 `multiprocessing`

### daemon=True

守护线程。当所有非守护线程结束时，守护线程被强制终止，程序退出。
适合"不重要"的后台工作（日志、心跳）。注意：不会执行清理操作。

## subprocess

从 Python 启动外部进程（shell 命令、其他程序）。与 threading 完全不同：
- `threading`：同一进程内的并发
- `subprocess`：启动一个全新的外部进程

### shell=True

通过系统 shell（`/bin/sh`）解释命令，支持管道 `|`、通配符 `*`、环境变量 `$HOME`、
命令链接 `&&` 等 shell 语法。不加 `shell=True` 则直接调用程序，命令必须是列表形式。

### run() vs Popen()

- `run()`：阻塞等待命令完成，返回结果。简单场景优先用。
- `Popen()`：立刻返回，进程在后台运行，可以流式读取输出。

## threading.Lock

`with self._lock:` 获取互斥锁，确保同一时间只有一个线程能执行块内代码。
离开 `with` 块时自动释放（即使抛异常也会释放）。

## s08 中 thread + subprocess 配合使用的原因

两者解决不同问题：
- `threading`：让主线程（agent 对话循环）不阻塞
- `subprocess`：在后台线程里实际执行外部命令

```
主线程 (agent loop)                后台线程
  │                                  │
  ├── 创建 Thread ──────────────────→│
  │                                  │ subprocess.run("pytest")
  │ 立刻返回 task_id                  │ 等待命令完成...
  │ 继续对话                          │
  │                                  │ 完成，结果放入 notification_queue
  │ drain_notifications() ←──────────┘
  │ 把结果注入对话
```

只用 subprocess 不用 threading → agent 阻塞，等命令跑完才能继续对话
只用 threading 不用 subprocess → 没法执行外部 shell 命令

组合使用 = 非阻塞地执行外部命令。
